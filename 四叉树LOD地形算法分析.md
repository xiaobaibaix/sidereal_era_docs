# 四叉树 LOD 地形算法分析

> 本文是对 CSDN 文章《回顾四叉树 LOD 地形（上）》（作者：灿VS自己 / lgcvsself，2014-05-27）的整理与分析。
> 算法主要参考潘李亮的四叉树 LOD 地形论文与曾涛的代码实现。

---

## 一、要解决的核心矛盾

地形渲染的根本问题是**三角形数量爆炸**。

- 边长 n 的高度图地形有 `2n²` 个三角形，全量提交给 GPU 是 `O(n²)`。
- 远处地形投影到屏幕后可能只占几个像素，甚至不可见，画了也看不出细节。

LOD（Level Of Detail，细节层级）的目标就是把"渲染精度"与"屏幕贡献"对齐：

- **近处密、远处疏**
- **视野外不画**（视锥体可见性剔除）

这套算法用 **四叉树 + 结点评价 + 标志数组** 三件套来实现。

---

## 二、实现 LOD 的三个难点

1. 以何种方式对地形进行简化。
2. 如何解决地形简化后出现的 **T 型裂缝**。
3. 视点移动时，地形细节的**突隐突现（popping）**：远处被简化的区域靠近后突然变详细，或近处详细区域远离后突然被简化，有光照时尤其明显。

> 作者当年因时间原因未处理第 3 点，改用**平滑（模糊）高度图**来缓解，同时还能减少三角形数量。前两点是本文重点。

---

## 三、地形的表示（理解全篇的钥匙）

### 1. 尺寸约束：(2ⁿ+1) × (2ⁿ+1) 个顶点

例如 9×9（n=3）。为什么要 +1？因为四叉树每次要把一块地形"四等分"，等分线要落在已有顶点上。`9×9 → 5×5 → 3×3 → 2×2` 的角点划分，每次中点都正好是整数索引的顶点。若是 8×8 就无法整齐对半分。

### 2. 基本单元是"三角形扇"，而不是独立三角形

一个地形结点 = **9 个顶点（4 角 + 4 边中点 + 1 中心）+ 10 个索引**，组成 8 个三角形围绕中心点呈扇形。

这是关键设计：**一个大三角形扇可以直接替换掉一片小三角形扇**——这正是"简化"的物理动作（前提是这块地形被评估为足够平坦）。

### 3. 分辨率级别（level）

- 最大的结点（4 角 + 4 边中点 + 中心）= 第 0 级。
- 每细分一次，子结点级别 +1。
- 最小的不可再分结点 = 最高级。

级别的作用：在预处理时用级别**快速算出结点中心顶点在顶点缓冲区中的索引**（每一级的步长是 `2^(maxLevel-level)`）。

因此结点在内部只用 `(中心顶点索引, 级别)` 两个数表示，**直到提交渲染时才展开成 10 个索引**。

## 四、简化流程：从递归到双队列

### 逻辑上是自顶向下的递归细化

注意方向：不是"由繁到简"，而是**由粗到精**。

```
评价(0级结点)
  需要分割 → 拆成4个子结点，各自递归
  不需要分割 → 直接送渲染
  视野外   → 忽略
```

递归到最小结点或结点不可分为止。

### 双队列消除递归

递归在大地形 + 实时场景里不理想（栈开销、剪枝控制难、cache 不友好）。潘李亮的做法是**双队列**：

```
队列A、队列B
把0级结点放入A
while A 或 B 非空:
    处理A中每个结点:
        需分割   → 4个子结点入B，自己出A
        不需分割 → 出A并送渲染
    A、B 交换身份
    重复
```

本质上这是**用 BFS（按层遍历）替换 DFS（深度递归）**。每一轮处理同一级别的所有结点，天然保证"逐级推进"，也让"控制相邻结点级别差"这件事更容易（同级结点是一批处理的）。

## 五、结点评价系统（决定分不分割）

算法的"大脑"，由两个因子合成。

### 因子 1：距离 / 尺寸（第一条评价公式）

```
L / D  与阈值 C 比较
```

- `L` = 视点到结点中心的距离
- `D` = 结点边长
- 当 `L/D > C` 时不分割

单独用距离不行：巨大但很远的结点如果不分割会丢大量细节；很小但很近的结点没必要再分。所以要**同时考虑距离和结点大小**，比值才有意义。

### 因子 2：粗糙度（第二条评价公式）

只看距离还不够。起伏剧烈的地形无论多远都希望能看出凹凸，于是引入粗糙度 `R`。

一个结点取 9 个高度误差（简化后的平面与真实地形面之间的差）：

- 4 条边中点误差 `dh1~dh4`
- 中心点误差 `dh0`
- 4 个子结点自身的粗糙度 `dh5~dh8`（递归定义，到叶子就忽略）

```
R = Max(dh0, …, dh8) / D
```

即**最大高度误差 / 边长**。`R` 越大越不该简化。第二条公式引入粗糙调节因子 `C2`。

### 合并

两条公式合成一个总判据。直觉是：

> 结点要满足"够远够小" **且** "够平坦"，才允许被简化（不分割）。

### 三个重要工程结论

1. **`C`、`C2` 越大 → 细节越多、三角形越多、开销越大**，需要在实测阶段调参。
2. **粗糙度可全部预处理**：`dh` 的递归计算放在加载阶段，存进一个和顶点缓冲区等大的数组，运行时直接查表，不影响实时性。
3. **平滑高度图能一石二鸟**：模糊后 `R` 降低 → 简化几率升高（三角形变少）+ "突隐突现"减轻（简化与真实的误差变小）。

## 六、修补 T 型裂缝

**问题**：相邻两块地形若分割级别相差太大，高分辨率一侧的边中点顶点会"悬空"在低分辨率一侧的边上，产生裂缝——有光照时特别明显。

### 第一步：控制相邻结点级别差 ≤ 1（用标志数组）

- 增设一个 `char` 标志数组，大小 = 顶点数，只存 0/1（0 = 对应结点不存在，1 = 存在）。
- 分割规则：
  - 结点**可分** → 自身标志设 1，**4 个子结点标志也设 1**。
  - 结点**不可分** → 自身标志设 1，**4 个子结点标志设 0**。
- 分割的**双重判据**：先过评价系统，**再**检查四周同级结点标志位是否都为 1。只要有一个邻居"不存在"（标志 0），当前结点就不许分割。
- 边缘结点：找不到的方向直接忽略；内部结点四个方向都要查。

> 示例：右侧结点的父结点先被判为不可分 → 右侧结点实际不存在 → 即使左侧结点通过了评价，查邻居时发现右邻居不存在 → 分割失败。这样把级别差锁死在 1 以内。

### 第二步：消除裂缝（在展开成 10 索引时做）

这是整个算法最精巧的地方：

- 结点展开成三角形扇（中心 + 8 三角形）时，检查**四周相邻同级结点的标志位**。
- 如果某方向邻居标志为 0（该方向是低分辨率的大结点），就**跳过这条边的中点顶点**。
- 于是原本 `0-3-4` 和 `0-4-5` 两个小三角形，合并成 `0-3-5` 一个大三角形，正好和低分辨率邻居的边贴合 → 高度差消失 → 裂缝消除。

一句话：**根据邻居标志位，动态决定四条边中点是否参与索引组装**。零额外内存、几乎零开销。

## 七、整体流程

```
每帧:
  清空标志数组
  0级结点入队列
  while 队列非空(双队列交替):
    for 队列中每个结点:
      if 结点在视锥外:
          忽略(可选：标志置0)
      elif 评价系统判定不需分割 OR 已是最小结点:
          标志置1, 子结点标志置0, 送渲染(展开索引时做裂缝消除)
      elif 四周邻居标志都为1:
          标志置1, 子结点标志置1, 4子结点入下一队列
      else:
          不能分割 → 送渲染
```

## 八、评价：优点、局限与现代视角

### 优点

- **数据结构极简**：结点只用"中心索引 + 级别"表示，标志数组只用 `char`，内存友好。
- **预处理 + 查表**：粗糙度离线算好，运行时零递归。
- **裂缝消除优雅**：不碰几何，只在索引组装阶段少放顶点，代价几乎为零。
- **评价函数物理意义清晰**：距离/尺寸 + 粗糙度，可调参数少（`C`、`C2`）。

### 局限

- **CPU 端逐结点评价、每帧重建**：典型的 2005 年前后 CPU LOD，地形极大时 CPU 成瓶颈。
- **popping 没根治**：靠平滑高度图缓解，未做 geomorphing（两级之间顶点插值过渡）。
- **顶点数据静态全量常驻**：`(2ⁿ+1)²` 顶点全在缓冲区，超大地形内存吃不消，无分块流式加载。
- **三角形扇图元效率一般**：现代 GPU 更喜欢 triangle strip / indexed list + 顶点 cache 优化。

### 现代视角

这套方法属于经典的 **CPU 端四叉树 LOD**，同类里更有名的是 **Röttger 的四叉树算法**（本文评价函数几乎就是 Röttger 那套 `L/D` + 粗糙度的思路）和 **ROAM**（二叉树三角剖分）。

现代大地形思路已经变了：

- **CDLOD / Geometry Clipmaps**：GPU 端做 LOD，用环形 clipmap 或网格 morph，天生解决 popping 和裂缝。
- **Chunked LOD**：地形切块，每块预生成多个 LOD 网格，配合流式加载支持超大世界。
- **裂缝处理**：现在多用 skirt（裙边）或 morph 到统一边界，而不是本文的边中点剔除。

本文价值在于——**把四叉树 LOD 的三大核心难点（简化策略、评价函数、T 型裂缝）用最朴素的方式讲透了**，是理解后续所有地形 LOD 算法的入门基石。
